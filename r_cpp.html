R

I have used R since 2003, where I picked it up for statistical computing (and applied statistics). While the core language has not changed too much since then, the R ecosystem (meaning: developer environments, language interoperability and extensions, packages and domain specific use-cases) has grown rapidly over the last few years.

R is considered to be an unconventional programming language. While most beginners often think of it as a fancy statistical calculator, it is both a powerful and flexible language / scripting language for the more general R environment (similar to how VBA is a scripting language for Excel). Although this definition downplays many of R strengths - it is accurate. For casual R users, much of the advanced mechanisms are not immediately obvious: many new R users learn how to write and use (user-defined) functions quite quickly, without having to know these details, for instance.

  • There are many guides to R on-line, and much of the basic syntax and functionality can be picked up using a combination of doing, reading the source code and (excellent) documentation of packages. A great book for newcomers to R is Learning R by Richard Cotton. For intermediate users, or those interested in the more advanced features of the language, Hadley Wickham’s Advanced R is hard to beat.

Performance: Integrated R & C++ via Rcpp

Rcpp is a R package enabling the easy use of compiled code within R. While the R environment makes extensive use of compiled C and Fortran, including compiled code in packages and user-defined functions was, until now, painful. It is difficult to overstate how workflow in R is extended by Rcpp. Along with the optimised matrix operations of the Armadillo C++ library, available to R through the RcppArmadillo package, and the recent RcppParallel package, leveraging CPU multi-core concurrency direct from compiled C++, R is a serious environment for numerical programming. A good introduction to RcppParallel is this talk.

A trivial example of how simple it is to deploy solutions in C++

#include <Rcpp.h>
using namespace Rcpp;

// [[Rcpp::plugins("cpp11")]]

// [[Rcpp::export]]
int i_sect(const IntegerVector& x, const IntegerVector& y ) {
  std::unordered_set<int> s(x.begin(), x.end());
  return std::count_if(y.begin(), y.end(), [&](int k) {return s.find(k) != s.end();});
}

// [[Rcpp::export]]
double i_prod(NumericVector& x, NumericVector& y) {
  return std::inner_product(x.begin(), x.end(), y.begin(),0.0);
}
require(microbenchmark)
x<-runif(100000)
microbenchmark(sum(x * x),i_prod(x,x))

Resulting in a 2x to 4x speed gain. While C++ provides performance benefits, making previously infeasible numerically intensive simulation tasks feasible, it also exposes data structures and algorithms to R.


Here is a simple template function to sort a vector relative to another. This is 2x as fast as the naive R method, and even faster for small vectors.

#include <Rcpp.h>
using namespace Rcpp;

// [[Rcpp::plugins("cpp11")]]

template <typename T, typename U> void sorty(T& a, U const x) {
  std::sort(a.begin(), a.end(),
            [&x](int i, int j) -> bool { return x[i] > x[j]; });
}

// [[Rcpp::export]]
NumericVector sort_by(NumericVector& to_sort, NumericVector& by){
  IntegerVector toOrder = seq_len( by.size() )-1; //similar to std::iota, gives {0,1,2,..,by.size()-1}
  sorty(toOrder,by);
  return(to_sort[toOrder]);
}
a<-rexp(1000,1)
b<-rnorm(1000,0,1)
microbenchmark(a[order(b,decreasing = TRUE)],sort_by(a,b))


Deployment: Shiny and Knitr

Data: tidyverse and data.table

Tidyverse is the term given to the suite of R packages that have significantly enhanced (to put it mildly) how one approaches the task of data analysis, data wrangling and data visualisation in R – compared to the base R workflow of data.frame and plot. While data frames are still a core feature of the language, and base R plot can plot almost anything (if you dedicate enough time to the task), the tidyverse philosophy significantly reduces the time (and effort) it takes to produce those interesting and complex analyses base R struggles with.

Data.table offers another replacement for base data frames, utilising by-reference operations on data, it is incredibly fast and efficient and can work with GB’s of data effortlessly. While dplyr (part of the tidyverse suite) offers a clean and consistent syntax - together with much improved performance over direct use of data.frame - data table should be considered where speed is important.
